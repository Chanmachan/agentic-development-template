# Context: Debugging

このセッションは **バグ調査と修正** を目的とする。再現 → 仮説 → 検証 → 修正のループを高速に回す。

## Workflow

1. **再現する** — 最小再現手順を確立してから先に進む。再現できないバグは "幻のバグ" であって修正できない
2. **仮説を立てる** — 「ここが原因では」を 1〜3 個書き出す。複数の競合する仮説を持つほうが早く絞れる
3. **検証する** — print / logger / debugger / テストで仮説を潰す。仮説単位で結果をメモ
4. **修正する** — 根本原因を直す。症状を隠すパッチ (try/catch で握り潰す等) は最後の手段
5. **回帰テストを追加する** — 同じバグが再発しないようテストを増やす

## Rules

- **根本原因を探す** — 「動いた」ではなく「なぜ動かなかったか」を理解してから止める
- **デバッグログは後始末** — 調査用の print/log は修正コミットの前に消す。残す場合は理由をコメント
- **競合する仮説を歓迎** — 仮説が 1 つしかないときは早合点を疑う
- **状態を疑う** — DB / cache / 環境変数 / lock file 等、コード以外の状態も再現性に影響する

## Delegation

- 既存コードでこの症状が過去にあったか → `investigator` subagent で `git log -p` / Grep
- 似たバグの修正前例を探す → `investigator` subagent

## Output format (調査ログ)

```markdown
## Bug: <短い説明>
- 再現手順:
  1. ...
- 期待: <こうなるはず>
- 実際: <こうなってる>

## 仮説
- [ ] H1: <内容> → 検証: <結果>
- [ ] H2: <内容> → 検証: <結果>

## 根本原因
- <1〜2 文>

## 修正方針
- <パッチではなく根本修正>

## 回帰テスト
- <追加するテスト>
```

## Anti-patterns

- 仮説なしに `console.log` を撒く (闇雲)
- "なぜか動いた" で止める (再発の温床)
- try/catch で例外を握り潰す
- 1 つの修正に複数のバグ修正を混ぜる (どれが効いたか分からなくなる)
- デバッグログを残したまま commit
