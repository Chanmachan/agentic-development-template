# Verify Guide 生成

指定した PR / ブランチの差分を読み、**(1) ユーザー視点で何ができるようになったか** と
**(2) 自分の手元でどう動作確認するか** を、コピペで実行できる runbook にまとめる。

レビュー前・マージ前に「で、これ何ができて、どう試すの?」を毎回聞かなくて済むようにするためのコマンド。

## 引数

$ARGUMENTS は任意:
- 数字 (`78`) → PR 番号
- URL (`https://github.com/org/repo/pull/78`) → 末尾の番号を抽出
- 空 → 現在のブランチ `HEAD` vs `main` の差分を対象

## 手順

1. **対象の差分を把握する**
   - PR 番号あり: `gh pr view <番号>`(title / body / base)+ `gh pr diff <番号>`
   - 空: `git branch --show-current` / `git log main..HEAD --oneline` / `git diff main...HEAD`
2. **ユーザー視点の「できるようになったこと」を抽出する**(実装詳細ではなく振る舞い)
   - 新規 / 変更された **API エンドポイント**、**画面 / ルート**、**ユーザーフロー**(ログイン、登録、CRUD 等)
   - mock → 実 API 配線なら「これまで固定表示だった X が実データで動く」と書く
   - 内部のみの変更(テスト / 型 / リファクタ / ドキュメント)で**ユーザー視点の変化が無い**場合は、その旨を明記する(「画面上の変化なし。確認はテスト / CLI で行う」)
3. **動作確認手順を組み立てる**(下記「リポジトリ既知ファクト」を使い、具体コマンドに落とす)
   - 必要なら **seed / フィクスチャ**の準備手順
   - **サーバ再起動の要否**を必ず判定する: API ルートやハンドラを変更した PR は、稼働中の dev サーバが古いコードのままのことがある(ホットリロード漏れ)。期待しない 404 や古い振る舞いが出たら stale のサイン。**再起動を手順の先頭近くに明記**する。
   - **ブラウザ手順**(ログイン → 対象 URL → 期待表示 / 操作)
   - **DB / CLI 確認**(変更が永続状態に出る場合)
   - read 専用 / 認証フローなら **curl での代替確認**も添える
4. **runbook を出力する**(下記フォーマット、日本語、コマンドは ```bash ブロックでコピペ可能に)

## 原則

- 実装の説明ではなく **「ユーザーが何を試せて、何が起きれば成功か」** を書く
- コマンドは **そのままコピペで動く**形(絶対パス・env・コンテナ exec 等)にする
- **サーバ再起動の要否**を必ず判定して明記する(API 変更 PR の最頻つまずきポイント)
- 確認に使う **ID や固定値を決め打ちしない**。seed 後に実値を引く手順を入れる
- 機能に**ユーザー視点の変化が無い**なら正直にそう書き、テスト / CLI での確認に誘導する
